일시: 2026년 4월 22일
참석자: PO(사용자), 10년차 PM, 15년차 CTO, 5년차 프론트엔드, 3년차 UI/UX, 2년차 백엔드, 1년차 QA

🗣️ 팀원별 의견 및 쟁점 사항

🧑‍💼 10년차 PM (Product Manager)

"반갑습니다. 이번 프로젝트의 핵심은 **'가장 간편한 텍스트 파일 리더'**입니다. 복잡한 가입 절차 없이 파일을 던져 넣으면 바로 읽을 수 있고, 기기를 바꿔도 이어서 볼 수 있어야 합니다.
MVP(최소 기능 제품) 단계에서는 철저히 '읽기 경험'과 '자동 동기화'에 집중하고, EPUB 지원이나 소셜 기능 등은 Phase 2로 미루는 것을 제안합니다. 오늘 회의로 명확한 Scope(범위)를 픽스하고 싶습니다."

🧙‍♂️ 15년차 CTO (Chief Technology Officer)

"현재 아키텍처는 React 기반 SPA(Single Page Application)에 Firebase(Auth, Firestore)를 붙이는 서버리스(Serverless) 구조로 방향을 잘 잡았습니다. 유지보수 비용을 0원에 가깝게 가져갈 수 있죠.
다만 성능적 관점에서 짚고 넘어가야 할 게 있습니다. 소설 텍스트 전체를 Firestore DB의 Document(문서) 하나에 통째로 저장하는 방식은 Firestore의 문서 당 1MB 제한에 걸릴 위험이 있습니다. 장편 소설 텍스트 파일은 1MB를 넘는 경우가 흔하거든요."

👩‍💻 2년차 백엔드 개발자

"CTO님 말씀에 동의합니다! 그래서 데이터 모델링을 수정했으면 합니다.

**메타데이터(제목, 읽은 위치, 날짜 등)**는 빠른 조회를 위해 Firestore에 저장하고,

실제 소설 텍스트(.txt) 내용은 용량 제한이 없는 Firebase Storage에 파일 형태로 올려서 URL만 Firestore에 연결하는 방식을 제안합니다."

👨‍🎨 3년차 UI/UX 디자이너

"현재 프로토타입 UI는 깔끔하지만, 모바일 환경에서의 '몰입감'을 더 끌어올려야 합니다.
사용자가 글을 읽기 시작하면 상단 헤더가 시야를 방해하지 않도록 스크롤 다운 시 헤더 숨김, 스크롤 업 시 헤더 노출 애니메이션이 필수입니다. 그리고 텍스트 뷰어의 생명은 '타이포그래피'입니다. 시스템 기본 폰트 외에 '리디바탕'이나 'KoPub바탕' 같은 가독성 높은 웹 폰트를 내장했으면 좋겠습니다."

🧑‍💻 5년차 프론트엔드 개발자

"UI/UX 디자이너님 의견 반영해서 헤더 동작 구현하겠습니다. 추가로 기술적인 쟁점이 하나 있습니다.
웹브라우저에서 긴 텍스트를 렌더링할 때 '무한 스크롤(위아래로 내리기)' 방식과 리디북스 같은 '페이지네이션(좌우로 넘기기)' 방식 중 어떤 것을 택할지 결정이 필요합니다. 모바일에서는 스와이프로 넘기는 페이지네이션이 선호도가 높지만, 웹 환경 개발 공수는 스크롤 방식이 훨씬 적습니다."

🕵️‍♀️ 1년차 QA (Quality Assurance)

"테스트 시나리오를 짜다 보니 몇 가지 엣지 케이스(예외 상황)가 우려됩니다.

인코딩 문제: 오래된 무협/판타지 소설 txt 파일은 UTF-8이 아니라 EUC-KR(ANSI)로 인코딩된 경우가 많습니다. 이걸 그대로 올리면 글자가 다 깨질 텐데, 브라우저에서 자동 인코딩 변환을 지원해야 할 것 같습니다.

익명 로그인의 휘발성: 비회원(익명) 동기화는 편하지만, 사용자가 브라우저 캐시나 쿠키를 날려버리면 익명 계정 UID가 바뀌어서 기존 서재 데이터에 영영 접근할 수 없게 됩니다. 이에 대한 안내나 대비책이 필요합니다."

🎯 [최종 결정 필요 사항] (PO님 결재 바랍니다)

팀원들의 의견을 종합하여, 명세서 작성을 위해 PO님께서 다음 4가지 항목을 결정해 주셔야 합니다.

Q1. 파일 저장 아키텍처 구조

[ ] A안: 텍스트 전문을 Firestore에 저장 (개발이 빠르지만 1MB 제한, 단편/중편 위주)

[ ] B안: 메타데이터는 Firestore, 텍스트 원본은 Storage에 분리 저장 (장편 가능, 백엔드 로직 약간 추가됨) [팀 추천]

Q2. 계정 정책 (익명 로그인 휘발성 대비)

[ ] A안: 극단적 심플함. 영구 익명 로그인 유지. (캐시 지우면 날아간다고 경고문만 표시)

[ ] B안: 익명 로그인으로 시작하되, 설정 메뉴에 "Google/이메일 계정 연동하여 데이터 영구 보존하기" 버튼 추가 [팀 추천]

Q3. 리더기 읽기 방식 (UI/UX)

[ ] A안: 스크롤 방식 (웹툰처럼 위아래로 스크롤, 구현이 가볍고 빠름)

[ ] B안: 페이지 방식 (책처럼 좌우 스와이프/클릭으로 넘김, 모바일 이북 리더기 감성)

[ ] C안: 스크롤을 기본으로 하되, 추후 설정에서 선택 가능하게 확장

Q4. 인코딩 (한글 깨짐) 대응

[ ] A안: UTF-8 파일만 지원 (사용자가 알아서 변환해서 올려야 함, EUC-KR 업로드 시 에러 메시지 띄움)

[ ] B안: Frontend에서 파일 업로드 시 jschardet 라이브러리 등을 사용해 EUC-KR을 감지하고 UTF-8로 자동 변환 처리 [팀 추천]
